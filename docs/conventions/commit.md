# 커밋 컨벤션

이 문서는 `/commit` 스킬과 모든 에이전트가 커밋 메시지를 작성할 때 따르는 규칙입니다.

---

## 기본 형식

```
<type>: <작업 내용>
```

단일 도메인/기능 범위의 작업인 경우:

```
<type>(<scope>): <작업 내용>
```

---

## 타입 목록

| 타입 | 사용 조건 |
|------|-----------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 코드 구조 개선 |
| `chore` | 빌드, 의존성, 기타 유지보수 |
| `setting` | 프로젝트 설정, 환경 구성 |
| `docs` | 문서 추가 및 수정 |
| `test` | 테스트 코드 추가 및 수정 |
| `style` | 포맷팅, 코드 스타일 (동작 변경 없음) |

---

## 스코프(scope) 규칙

- 스코프는 `docs/domains.md`에 등록된 대분류 목록에서 선택한다.
- 변경 사항이 **단일 도메인이나 하나의 기능 범위**에 집중될 때만 사용한다.
- 여러 도메인에 걸친 작업은 스코프 없이 작성한다.

---

## 커밋 분리 원칙

- 서로 관계없는 기능, 설정, 문서 변경이 함께 존재하면 자동으로 기능별 커밋으로 나눈다.
- 한 커밋은 하나의 목적만 가져야 한다. 예를 들어 Swagger 문서화, H2 설정, 하네스 문서 추가는 각각 별도 커밋 대상이다.
- 커밋 전 `git diff`를 보고 변경사항을 목적별로 분류한 뒤, 관련 파일만 선별적으로 스테이징한다.
- 같은 파일 안에 여러 목적의 변경이 섞여 있어 분리하기 어렵다면 예외적으로 한 커밋에 포함할 수 있다. 이 경우 더 중심이 되는 변경 목적의 타입과 메시지를 사용한다.
- 사용자가 단순히 "커밋해줘"라고 요청해도, 변경사항이 여러 독립 목적이면 에이전트가 알아서 여러 커밋으로 나눈다.

---

## 예시

```
setting: 하네스 엔지니어링 초기 구조 셋업
feat(user): 회원가입 API 엔드포인트 추가
fix(order): 주문 금액 계산 오류 수정
refactor(auth): JWT 토큰 검증 로직 분리
docs: 커밋 컨벤션 및 도메인 분류 문서 추가
chore: postgresql 의존성 추가
```

---

## 메시지 작성 원칙

- 한국어로 작성한다.
- 현재형으로 작성한다 ("추가했다" → "추가").
- 무엇을 했는지보다 **왜** 했는지가 명확하면 본문에 추가한다.
- 본문이 필요한 경우 제목과 한 줄 띄고 작성한다.
