# agents.md

이 파일은 Claude Code 하네스에서 에이전트가 어떻게 역할을 나누고 협업하는지 정의합니다.
모든 에이전트는 `CLAUDE.md`의 행동 원칙을 기본으로 따릅니다.
백엔드 팀형 역할 정의와 요청별 분배 기준은 `docs/harness/agent-team.md`를 우선 참고합니다.

---

## 에이전트 사용 원칙

- 단일 조회나 간단한 작업은 에이전트를 생성하지 않고 직접 처리한다.
- 에이전트는 **독립적으로 실행 가능한 작업**에만 사용한다 — 결과를 받기 전에 다음 단계가 결정될 수 없는 경우는 순차 처리한다.
- 에이전트에게 위임할 때는 컨텍스트를 명시적으로 전달한다. 에이전트는 대화 맥락을 모른다.
- 에이전트의 결과는 신뢰하되 검증하라 — 의도한 대로 했는지 실제 변경사항을 확인하라.

---

## 에이전트 역할 정의

프로젝트 운영 역할은 `docs/harness/agent-team.md`에 정의된 팀 구성을 따른다.
Claude Code 하네스에서 실제 사용 가능한 에이전트 타입은 아래 역할로 매핑한다.

- Backend Developer, DB Manager, DevOps Config Engineer → `general-purpose` 또는 구현 가능한 작업 에이전트
- QA Engineer, Documentation Manager, API Designer → `general-purpose` 또는 분석 전용 에이전트
- 파일/심볼 위치 탐색 → `Explore`
- 여러 파일 변경 전략 수립 → `Plan`

### Explore (탐색 에이전트)

**목적**: 코드베이스 내 파일, 심볼, 패턴 위치 파악

**사용 조건**:
- 특정 파일이나 심볼의 위치를 모를 때
- 3회 이상의 검색이 필요한 광범위한 탐색일 때

**사용하지 않는 경우**:
- 파일 경로를 이미 알고 있는 경우 → `Read` 직접 사용
- 특정 문자열 검색 → `grep` 직접 사용
- 코드 리뷰, 설계 분석, 교차 파일 일관성 검사 (전체 파일을 읽지 않으므로 부정확함)

---

### Plan (설계 에이전트)

**목적**: 구현 전략 설계 및 단계별 계획 수립

**사용 조건**:
- 여러 파일에 걸친 비자명적 변경이 필요할 때
- 아키텍처 트레이드오프 검토가 필요할 때

**출력 형식**:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```

---

### general-purpose (범용 에이전트)

**목적**: 복잡한 멀티스텝 리서치 및 코드 실행

**사용 조건**:
- 처음 몇 번의 시도로 올바른 결과를 찾기 어려운 열린 질문
- 독립적으로 병렬 처리할 수 있는 작업

---

## 병렬 실행 규칙

독립적인 작업은 동일한 응답에서 여러 에이전트를 병렬 호출한다:

```
# 올바른 예: 두 탐색이 서로 의존하지 않음
Agent(탐색: "UserService 위치") + Agent(탐색: "OrderService 위치")

# 잘못된 예: 첫 번째 결과가 두 번째 입력이 됨
Agent(탐색: "파일 찾기") → 결과 확인 후 → Agent(수정: 해당 파일)
```

---

## 에이전트 프롬프트 작성 지침

에이전트는 대화 컨텍스트 없이 콜드 스타트된다. 프롬프트에 반드시 포함할 것:

1. **목적**: 무엇을 달성하려 하는가
2. **배경**: 관련 파일 경로, 이미 확인한 사항, 배제한 접근법
3. **출력 형식**: 짧은 요약인지, 파일 변경인지, 분석 보고서인지
4. **범위**: 리서치만인지, 코드 작성까지인지 명시
